# tasktab 项目进度

> 这份是给不懂代码工程的人看的项目进度。它只回答四件事：现在做到哪了、已经完成什么、下一步做什么、有什么风险。

项目定位：**TaskBoard**——一个 Mac 桌面应用，把你所有项目的进度集中在一个看板上。你不用手动更新看板，干活时随手更新各项目的进度文件，看板自动跟着秒级刷新。

## 已完成

### 2026-06-14：架构大调整——彻底取消"进度文件"，看板只认项目三件套（重要）

- **这是 James 提出的一次架构简化，目的是去掉冗余。** 原本每个项目要单独维护一个 `PROGRESS.md` 进度文件（靠 `/kanban` 工具写），同时又有三件套（AGENTS/INDEX/CHANGELOG）。两套并存、信息重复。
- **改成什么样**：`PROGRESS.md` 整个废掉，看板需要的所有信息都从三件套里读——
  - 项目状态（进行中/暂停/完成）和一句话描述 → 写在 AGENTS.md 顶部
  - 整体进度 → 数 CHANGELOG.md「项目阶段」清单里勾了几项算出来（勾 3/4 就是 75%）
  - 接下来做什么 / 被什么卡住 → INDEX.md 的「当前接力点」
  - 最后更新日期 → CHANGELOG 最新一条的日期
- **连带退役**：`/kanban` 这个专门写进度文件的工具下岗了（已从所有 AI 助手卸载、归档留底）；它的活由两个工具接手——日常收尾用 `/wrap-up`（顺手把状态、阶段勾选更新好），首次接入看板或重铺信息用 `/outkanban`（一次性把全套字段生成好）。登记项目仍用 `cra add`，但它不再生成任何文件。
- **对项目的意义**：项目维护从"两套真相"变成"一套真相"——你只管写三件套，看板自动跟着变，再没有"忘了同步 PROGRESS"的问题。这是把"文件是唯一真相"理念贯彻得更彻底的一步。
- **怎么做到不出乱子**：保留了完整的防御性降级——任何项目没填这些字段，卡片只显示「未接入看板」，绝不崩。后端 37 项自动化测试全过，前端构建通过。还顺手修了一个跨午夜会让日期差一天的老 bug。
- **怎么验收**：给某个项目用 `/outkanban` 生成看板字段（或手动在三件套里填），点进详情页看状态/进度/简介/架构图是否正确；改 CHANGELOG 勾掉一个阶段，看板进度条应 1 秒内变化。

### 2026-06-13：详情页改版——看板开始"读懂"项目三件套（新功能）

- 这是本次规划之外、James 现场提的产品改进，分两块：
- **卡片首页**：原来卡片中间显示"阶段 1/1·需求与架构"这种内部术语，现在换成**一句话项目描述**（比如"多账号一键切换工具"），普通人扫一眼就知道这项目是干嘛的。描述由各项目自己在进度文件里写一行 `desc` 提供，没写就不显示、不影响其它内容。
- **详情页**：从"只展示一个进度文件"升级成**读取项目三件套的结构化信息**——项目简介（一段话介绍）、项目阶段清单（带"做完打勾/没做空心圈"的状态和每项描述、超过十项可滚动）、架构图（用文字描述的流程图会被**画成真正的图**）。同时去掉了原来偏技术的整体进度文字、阶段时间线和进度文件正文预览，按钮也从"打开 PROGRESS.md"改成"打开 INDEX.md"。
- **怎么做到不乱**：看板依然零智能，只是按约定好的标题（## 项目简介 / ## 架构图 / ## 项目阶段）去三件套文件里"对号入座"地取内容；任何一块没写，详情页对应区域就不显示，绝不报错。这套约定已写进权威设计文档（02 §1.1b）和自动写进度的插件规则里，新建项目的模板也预留了可选位置。
- **对项目的意义**：看板的价值从"显示进度条"提升到"一页看懂一个项目"——是什么、做到哪了、长什么结构，都在详情页一目了然。而且把"内部术语"挡在了非工程读者视线之外。
- **怎么验收**：后端逻辑（提取简介/阶段/架构图三块）已加自动化测试并全部通过，前端类型检查、整体构建（含架构图绘图库按需加载）都跑通。真机效果需要 James 给某个项目的三件套补上这三块内容后，点进详情页查看。

### 2026-06-13：测试稳定性修复（重要）

- 复查代码时发现后端有一个测试"有时通过、有时失败"——这种不稳定的测试最危险，因为它会让人误以为代码坏了，或反过来掩盖真正的问题。
- 根因：有一个测试在运行时偷偷改了"用户主目录"这个全局设置，而后端测试是同时并行跑的，这一改就会干扰到当时正在跑的其它测试，结果就时好时坏。
- 已修复：把涉及主目录的逻辑改成"由调用方明确传入"，测试不再去动全局设置。修完连续并行跑 5 次，34 项测试全部稳定通过。
- **对项目的意义**：之前进度页写的"测试全过"现在才真正站得住脚——自动化测试是可信的安全网，后续真机验证若出问题，可以放心排除"是不是测试本身不靠谱"这一项。

## 风险与注意

| 风险 | 当前处理 |
|---|---|
| 数据格式（进度文件 / 登记表的字段）一旦定错，后面全得返工 | 已约定：格式以 02 文档为唯一权威，先把契约定死再写代码 |
| 看板若"自作聪明"去理解文件，会变慢且易错 | 设计铁律：看板零智能，只做确定性读取，绝不调用 AI |
| 进度文件损坏可能让看板崩溃 | 要求解析必须防御性，坏文件只显示"格式异常"卡片，不影响其他项目 |
| 进度只写技术细节，非工程读者看不懂 | 每次同步必须用普通中文解释"这一步意味着什么" |

## 每阶段同步要求

每完成一个里程碑或一个可验收小任务，agent 必须同步本文件：

- **已完成**：用非工程语言说明本阶段交付了什么。
- **下一步**：说明下一个里程碑要做什么，为什么先做它。
- **验收方式**：说明用户怎么判断这一步真的完成（对应 02 文档里每个里程碑的验收清单）。
- **风险变化**：新增、降低或仍未解决的风险。

不要只写代码文件名和命令；如果必须提命令，要用一句话解释它对项目进度意味着什么。
