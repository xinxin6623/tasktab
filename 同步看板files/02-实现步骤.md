## 0. 总览

构建一个 macOS 任务看板应用 TaskBoard,由四个组件构成:

1. **数据契约**:`PROGRESS.md`(每项目一份)+ `registry.yaml`(全局登记表)
2. **`cra` CLI**:Python 脚本,负责项目登记/移除/生成模板
3. **TaskBoard 桌面应用**:Tauri 2,读 registry → 解析各 PROGRESS.md → 渲染看板,FSEvents 监听自动刷新
4. **`progress-tracker` skill**:让 Claude Code 在干活时自动更新 PROGRESS.md(见文档 03)

设计铁律:
- **App 端零智能**。只做确定性解析与渲染,不调用任何 LLM。
- **文件是唯一真相**。App 不持有独立状态,所有写操作落到 registry.yaml,所有读操作来自文件系统。
- **解析必须防御性**。frontmatter 缺失/损坏时降级显示,绝不崩溃。

## 1. 数据契约(先于一切代码确定)

### 1.1 PROGRESS.md frontmatter schema

```yaml
---
project: voice-pipeline        # 唯一 id,与 registry 对应,kebab-case
desc: 实时语音对话 pipeline    # 可选,项目一句话描述,卡片展示用,缺省按空串处理(≤30 字,卡片最多两行)
status: active                 # active | paused | done
stages:                        # 项目全程的阶段规划,有序列表
  - 需求与架构
  - ASR 接入
  - barge-in 状态机
  - 联调打包
current_stage: 3               # 1-based,指向 stages 中的当前阶段
stage_progress: 60             # 0-100,当前阶段内进度,可选,缺省按 0 处理
next:                          # 接下来要做的事,有序
  - 打断信号去抖逻辑
  - 写集成测试
blocked_by: []                 # 可选,阻塞项描述列表
updated: 2026-06-13            # ISO date,最后更新日期
---

## 阶段记录

(以下为自由 markdown 正文,App 仅做只读预览渲染,不解析)
```

解析规则:
- 整体进度 = `(current_stage - 1 + stage_progress/100) / len(stages)`,status 为 `done` 时强制 100%
- `current_stage` 越界、`stages` 为空、frontmatter 缺失或 YAML 解析失败 → 该项目卡片显示「⚠ 格式异常」+ 打开文件按钮,其余项目正常渲染
- frontmatter 中出现 schema 之外的字段一律忽略(向前兼容)

### 1.1b 详情页三件套块(App 从 INDEX.md / CHANGELOG.md 提取)

详情页除 PROGRESS.md 外,还从**项目根固定文件名** `INDEX.md` / `CHANGELOG.md` 提取三个约定块。
全部块**缺失即静默降级**(该区域不显示),绝不崩溃——延续防御性解析原则。App 端零智能,只做确定性 markdown 块定位与文本截取。

**INDEX.md `## 项目简介`**:标题到下个 `##` 之间的纯文本 → 详情页阶段表上方的简介段(建议 50-70 字)。

**INDEX.md `## 架构图`**:其下首个 ```mermaid 代码块 → 详情页用 mermaid 本地渲染(动态 import,零网络)。渲染失败降级显示原文。

```markdown
## 架构图
​```mermaid
flowchart TD
  A[入口] --> B[处理]
​```
```

**CHANGELOG.md `## 项目阶段`**:其下 markdown checkbox 列表 → 详情页阶段分块列表(10 行+滚动)。
每行 `- [x]/[ ] <阶段名> — <描述>`:`[x]`/`[X]`=完成,描述用 ` — ` / ` - ` 分隔(可选)。块下方的变更流水不解析。

```markdown
## 项目阶段
- [x] 需求与架构 — 明确方案
- [ ] 联调验收
```

> 卡片首页"一句话描述"由 PROGRESS.md frontmatter `desc` 提供(见 §1.1);它与 INDEX `## 项目简介` 配套:desc 是 20-30 字一句话(卡片用),简介是 50-70 字一段(详情页用)。

### 1.2 registry.yaml schema

路径:`~/.ai-vault/taskboard/registry.yaml`

```yaml
version: 1
projects:
  - id: voice-pipeline           # 与 PROGRESS.md 的 project 字段一致
    name: 语音管线               # 看板显示名
    path: /Users/james/dev/voice-pipeline   # 项目根目录,绝对路径
    progress_file: PROGRESS.md   # 相对 path 的进度文件路径
    pinned: false                # 置顶
    added: 2026-06-13
```

- `cra` 与 App 的增删操作都读写此文件,写入必须原子(写临时文件后 rename)
- `path` 不存在或 `progress_file` 缺失 → 卡片显示「⚠ 文件缺失」,不崩溃

## 2. 技术栈与仓库结构

- **App**: Tauri 2 + Rust(后端:文件读写、`notify` crate 做 FSEvents 监听、shell 调用)+ 前端 Vue 3 或 React + Vite(选脚手架默认即可),TypeScript
- **Markdown/YAML 解析**:前端用 `gray-matter` + `marked`(或在 Rust 侧用 `serde_yaml` 解析后传结构化数据给前端——推荐后者,前端只管渲染)
- **CLI**: Python 3.12 + `uv`,单文件优先,依赖仅 `pyyaml` + `click`
- **不引入数据库**

```
taskboard/
├── app/                  # Tauri 2 项目
│   ├── src/              # 前端
│   ├── src-tauri/        # Rust 后端
│   └── ...
├── cli/
│   └── cra.py            # 登记 CLI
├── skill/
│   └── progress-tracker/ # 按文档 03 生成
│       └── SKILL.md
├── scripts/
│   └── install.sh        # 软链 cra 到 ~/bin、安装 skill 到 ~/.claude/skills/
└── docs/                 # 本套文档归档于此
```

## 3. 里程碑与验收标准

按顺序执行,每个里程碑完成后运行其验收项,全部通过再进入下一个。

### M1 — 数据层 + cra CLI

实现 `cli/cra.py`:

- `cra add <path> [--name NAME]`:校验路径 → 若无 PROGRESS.md 则按 1.1 schema 生成模板(project id 取目录名 kebab-case,stages 留一个占位阶段)→ 写入 registry(已存在同 path 则报错提示)
- `cra remove <id>`:从 registry 移除登记,不触碰项目文件
- `cra list`:终端表格输出所有项目及整体进度百分比
- registry 文件不存在时自动初始化(含 `version: 1`)

验收:
- [ ] 对一个空目录执行 `cra add`,生成合法 PROGRESS.md 且 registry 出现对应条目
- [ ] 重复 add 同一路径报错且 registry 无重复
- [ ] `cra list` 能对手工填写的多阶段 PROGRESS.md 算出正确百分比
- [ ] registry 写入是原子的(代码审查确认 temp+rename)

### M2 — App 骨架 + 仪表盘(只读)

- Tauri Rust 侧实现 command:`load_board()` → 读 registry、逐项目解析 PROGRESS.md frontmatter、返回结构化 JSON(含每项目的解析错误标记)
- 前端仪表盘:项目卡片网格。每卡:名称、整体进度条、`阶段 current/total · 当前阶段名`、next 前两条、updated、status 徽标
- 顶部汇总条:active / paused / done 计数;done 项目排末尾,pinned 置顶
- UI 风格见第 4 节,**在写任何前端代码前先读 frontend-design 相关规范(若环境提供)**

验收:
- [ ] 用 3 个真实项目 + 1 个故意写坏 frontmatter 的项目测试:3 个正常渲染,坏的显示「格式异常」卡片,App 不崩溃
- [ ] 进度条数值与 1.1 公式一致

### M3 — 详情页 + 动作按钮 + 增删

- 详情页:阶段垂直时间线(完成✓/当前高亮+阶段内进度/未来灰)、完整 next 列表、blocked_by 警示条(若非空)、正文 markdown 只读预览
- 动作按钮(Rust command 调 shell):
  - 「打开 PROGRESS.md」→ `open <file>`
  - 「VS Code 打开项目」→ `code <project_path>`(`code` 不在 PATH 时降级为 `open -a "Visual Studio Code" <path>`,再失败则 toast 提示)
- App 内添加项目:选择目录对话框 → 复用 cra 同等逻辑(在 Rust 侧重新实现或调用 cra.py,二选一,推荐 Rust 侧实现以免依赖 Python 环境)
- App 内删除:确认弹窗,文案明确「仅从看板移除,不删除任何文件」

验收:
- [ ] 两个按钮在真实项目上可用
- [ ] App 内添加的项目,`cra list` 能看到(双向一致)
- [ ] 删除后项目文件完好

### M4 — 文件监听自动刷新

- Rust 侧用 `notify` crate(macOS 后端即 FSEvents)监听:registry.yaml + 每个已登记项目的 PROGRESS.md
- 变更事件 debounce 500ms 后向前端 emit 增量刷新事件
- registry 变更时重建监听列表

验收:
- [ ] 终端 `vim` 修改某 PROGRESS.md 的 `stage_progress` 并保存,1 秒内看板进度条变化,无需任何手动操作
- [ ] `cra add` 新项目后,看板自动出现新卡片(无需重启 App)

### M5 — skill 安装 + 端到端验证

- 按 `03-SKILL创建规则.md` 生成 `skill/progress-tracker/SKILL.md`
- `scripts/install.sh`:软链 cra、复制 skill 到 `~/.claude/skills/progress-tracker/`、构建并安装 .app

验收(端到端):
- [ ] 在一个已登记项目中启动 Claude Code 执行一个小任务并要求其收尾,确认 PROGRESS.md frontmatter 被正确更新且 TaskBoard 自动反映

## 4. UI 规范(Claude 风格)

- 背景:暖米白 `#FAF9F5`;卡片:白色、`1px` 极浅暖灰描边、`12px` 圆角、hover 时轻微阴影
- 文字:深暖灰 `#3D3929`;次要信息 `#73726C`
- 强调色:珊瑚 `#D97757`(进度条填充、当前阶段高亮、主按钮)
- status 徽标:active 用珊瑚、paused 用暖黄 `#C2A24B`、done 用灰绿 `#6A8A6E`,均为浅色底深色字
- 标题用衬线字体(Georgia / 思源宋体栈),正文系统无衬线;中文字体栈需兜底 `PingFang SC`
- 密度:信息密集但留白充分,卡片网格 `minmax(280px, 1fr)`;支持窗口缩到单列仍可用
- 禁止:重阴影、渐变按钮、彩色背景块堆砌
