# TaskBoard 大白话说明书

> 这份文档给人看。讲清楚这个东西是什么、怎么用、为什么这样设计。

## 这是个什么东西

一个 Mac 桌面应用,名字叫 **TaskBoard**。它是你所有项目的统一进度看板:

- 每个项目自己维护一份 `PROGRESS.md` 进度文件(平时就是 Claude Code 在干活时顺手更新的)
- TaskBoard 监听这些文件,文件一变,看板秒级刷新——**你不需要手动更新看板,永远不需要**
- 打开看板,一眼看到:每个项目走到第几阶段、当前阶段干到哪了、接下来要干什么

核心理念一句话:**看板零智能,只读文件**。"自动更新"不是看板聪明,而是干活的人(Claude Code)按约定写文件,看板老老实实读。

## 日常怎么用

### 新项目接入(一次性,30 秒)

```bash
cd ~/dev/my-new-project
cra add . --name "我的新项目"
```

`cra` 会:
1. 在项目根目录生成一份模板 `PROGRESS.md`(如果还没有)
2. 把项目登记到 `~/.ai-vault/taskboard/registry.yaml`

之后 TaskBoard 里就能看到这个项目了。

### 平时推进任务

什么都不用做。你在项目里跑 Claude Code,它有一个 `progress-tracker` skill,每次任务收尾会自动更新 `PROGRESS.md` 的结构化字段(当前阶段、进度、下一步)。看板自动跟着变。

如果哪天想手动改进度——直接编辑 `PROGRESS.md` 的 frontmatter 就行,看板照样秒刷。文件就是唯一真相。

### 看板上能干什么

- **仪表盘**:所有项目卡片一屏看完。每张卡有进度条(阶段 3/4)、当前阶段名、下一步前两条、最后更新时间。顶部有 active / paused / done 计数
- **点进项目详情**:完整的阶段时间线(走完的打钩、当前的高亮、没到的灰着),完整的下一步清单,`PROGRESS.md` 正文预览
- **两个快捷按钮**:
  - 📄 打开 `PROGRESS.md` 看细节(系统默认编辑器)
  - ⌨️ 用 VS Code 打开整个项目
- **添加 / 删除项目**:界面里也能操作,和 `cra` 命令改的是同一个 registry 文件,双向一致
- **删除只是"从看板移除"**,绝不动你项目里的任何文件

## 多机同步怎么回事

**第一阶段(现在):靠 git,零额外设施。**
registry 在 `~/.ai-vault/`(本来就是 git 仓库),各项目的 `PROGRESS.md` 在各自项目仓库里。MacBook 和 Mac mini 各跑一个 TaskBoard,数据随你已有的 git 同步流动。

**第二阶段(以后想要再做):Mac mini 跑一个只读网页版。**
因为文件本来就同步到 mini 上了,服务端只是读本地文件渲染网页,手机浏览器也能看进度。实现成本很低,但不是现在的事。

## 为什么是这套设计

| 决策 | 理由 |
|---|---|
| 进度存在项目自己的 markdown 里,不存数据库 | 文件随项目走、随 git 同步、人和 AI 都能直接编辑,坏了能用任何编辑器修 |
| 约定 frontmatter 格式,而不是让看板"理解"任意 markdown | 解析是确定性代码,毫秒级、零成本、不会出错;智能放在写入端(Claude Code skill) |
| Tauri 做桌面应用 | 原生 .app、体积小、常驻不吃资源;界面用 web 技术写,Claude 风格好调 |
| registry 放 ai-vault | 蹭现有的 git 同步,多机问题直接消失 |

## 文件都在哪

| 东西 | 位置 |
|---|---|
| 项目登记表 | `~/.ai-vault/taskboard/registry.yaml` |
| 每个项目的进度 | `<项目根目录>/PROGRESS.md` |
| cra 命令 | TaskBoard 仓库内,安装时软链到 `~/bin/cra` |
| 自动更新 skill | `~/.claude/skills/progress-tracker/` |
| TaskBoard 应用 | `/Applications/TaskBoard.app` |
